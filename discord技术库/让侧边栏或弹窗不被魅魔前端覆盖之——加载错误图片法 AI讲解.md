第一步：故意设置一个“诱饵”
我们在代码的最后放了一个 <img> 标签：
        
    <img src="x" style="display:none;" onerror="...">

      
        src="x": 我们让这个图片标签去加载一个叫 x 的图片。很明显，这是一个无效的、根本不存在的图片地址。浏览器去加载它，100%会失败。

    第二步：设置一个“触发器”

        onerror="...": 这个属性就是我们的核心触发器。它的意思是：“嘿，浏览器！如果你加载 src 里的东西失败了，就请立刻执行我引号里的这些 JavaScript 代码！”

    第三步：把所有功能代码“塞进去”

        onerror="(function(){ ... })()": 我们把所有控制侧边栏功能的 JavaScript 代码，全都写在了 onerror 的引号里面。

    所以整个流程就是：

        浏览器读到 <img> 标签。

        尝试加载图片 x，失败了。

        “噢，失败了！”，于是立即触发 onerror 事件。

        浏览器开始执行 onerror 属性里我们写好的所有 JavaScript 代码。

        这些代码一旦执行，我们定义的 sT()、sB()、startScroll() 等所有函数就全部准备就绪了，侧边栏上的按钮也就“活”了过来。

三、为什么它能让侧边栏“独立”？

这正是这个技巧最妙的地方。

    自给自足，不依赖外部文件：我们没有一个单独的 .js 文件，也不需要在页面的其他地方放一个 <script> 标签。所有让侧边栏工作的 样式（CSS）、结构（HTML） 和 功能（JavaScript） 都被封装在了这一个代码块里。

    高内聚，可移植性强：你可以把从 <style> 开始，到最后那个 <img> 结束的整个代码块，看作一个“即插即用的独立组件”。就像一个U盘一样，你把它复制粘贴到任何支持HTML的地方，它自己就能独立运行，完全不需要依赖页面的其他部分。

    隐蔽执行：我们还给图片加了 style="display:none;"，这样即便是加载失败，页面上也不会显示一个难看的“破图”图标。整个过程对用户来说是完全透明的，他们根本不知道背后有一个“不存在的图片”在默默地为他们启动了所有功能。
