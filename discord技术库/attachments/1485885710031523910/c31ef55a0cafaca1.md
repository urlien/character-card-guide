# README

这份文档是给小白看的上手说明，帮助你先读懂 [AGENT.md]在讲什么。

你可以把这个项目理解成一种“在现成聊天网页里插入独立 UI”的做法。平台原本已经有消息区、输入框、顶部栏这些基础网页结构，我们不是重做整页网页，而是在这个现成页面里接入你想要的界面，尤其是独立于回复内容的界面，比如侧边栏、地图、弹窗、图鉴等，并将最新回复中的内容同步显示在这些界面中。

[AGENT.md] 不是具体代码模板，也不是某一个固定界面的成品。它更像是一份总规则，告诉你这套 UI 应该怎么拆、什么时候初始化、每次新回复来了以后应该怎么更新界面。

如果你是第一次接触这个体系，先记住一句话就够了：先把界面本身放好，再把回复里的数据投放进去。不要一上来就把“界面、样式、初始化、回复逻辑、正则替换”全混在一起写。

## 这套体系到底在做什么

它的核心目标，是在平台原生网页中实现“界面注入 + 回复投放”。

“界面注入”可以理解成：先在原页面里放一个长期存在的独立 UI 外壳。

“回复投放”可以理解成：当 AI 产生新回复后，把这条回复里的结构化数据提取出来，更新到那个独立 UI 里。

所以，这套方法不是让 AI 直接输出一大坨复杂 HTML 去撑起整个界面，而是让 AI 提供可解析的数据，再由前端代码负责真正的显示。

## 五类文件

整套实现固定拆成 5 类文件。你可以先把它们想成 5 个分工明确的角色。

### 1. 界面文件

它负责“长什么样的结构”，也就是界面骨架。

比如：外层容器、标题区、内容区、按钮区、隐藏状态区，这些都属于界面文件该负责的东西。

它不负责：

- 不负责写样式细节
- 不负责处理数据
- 不负责决定某条回复怎么更新界面

### 2. CSS 文件

它负责“好不好看”，也就是样式、颜色、排版、边距、层级、响应式布局。

它不负责：

- 不负责解析 AI 回复
- 不负责执行页面逻辑
- 不负责在 JS 里拼接数据

### 3. 初始化代码文件

它负责“页面刚打开时先做哪些准备”。

比如：创建全局变量、创建全局控制对象、处理界面定位、设置初始化完成标记。

它只运行一次。

它不负责：

- 不负责处理某一条具体回复的数据
- 不负责每来一条新消息就重新跑一遍

### 4. 回复代码文件

它负责“每次有新回复时，怎么把这条回复的数据投到界面里”。

比如：找到当前触发节点、确认自己是不是最新的回复、读取同级中间态、清空旧数据、渲染新数据。

它不负责：

- 不负责页面首次总初始化
- 不负责定义正则替换规则

### 5. 中间态正则列表文件

“中间态”可以理解成“介于原始文本和最终界面之间的过渡结构”。

这个文件负责把 AI 的原始回复，先变成更容易读取的结构化节点，比如隐藏 `div`、`data-*` 属性、同级数据块等。

它不负责：

- 不负责初始化页面
- 不负责真正把数据渲染到 UI
- 不负责写大段运行逻辑

## 最核心的运行流程

你可以把整套运行过程想成下面这 7 步：

1. 页面打开后，平台先把原来的网页内容加载出来。
2. 初始化 `img` 触发一次启动代码，做全局准备。
3. 独立 UI 的根结构准备好，页面里开始有一个长期存在的界面外壳。
4. 页面打开时，页面中的历史回复会一起出现，对应的多个回复 `img` 也可能一起加载。
5. 这时回复代码要先判断：自己是不是最后一个回复触发节点。如果不是，就直接退出。
6. 新回复生成后，只有最新回复中的回复 `img` 会加载，其必定是最后一个回复节点。
7. 真正允许继续运行的那个回复节点，再去读取自己父容器里的同级中间态。
8. 读取到数据后，清掉旧内容，把新数据投放到独立 UI 里。

这也是为什么 [AGENT.md] 一直强调两件事：

- 初始化只做一次
- 回复只处理最新节点

因为页面刚打开时，旧回复很多，多个回复触发器可能一起跑。如果不限制“只有最后一个能继续”，旧数据就可能盖掉新数据。

## 你该怎么开始做一个页面 DIY

如果你想做一个自己的页面，建议按这个顺序来，不要跳步。

### 第一步：先想清楚你要做什么类型的 UI

你要做的是状态栏、地图、日志面板、卡片列表，还是别的东西？先把目标想清楚，不要还没确定界面类型就直接写代码。

### 第二步：先画或整理界面结构

哪怕只是手画草图也可以。你至少要知道：

- 外层容器长什么样
- 主展示区在哪
- 控制区在哪
- 哪些数据需要隐藏存放

### 第三步：先决定数据分工

想清楚哪些数据要直接显示，哪些数据只作为隐藏状态保留，哪些数据只是当前这一轮临时使用。

这一步很重要，因为它会直接影响你后面的中间态设计和回复代码写法。

### 第四步：写初始化代码

把页面定位、全局对象、初始化标记这些准备工作做好。记住：初始化代码只跑一次。

### 第五步：写中间态规则

先把 AI 的原始文本转成容易抓取的结构。不要急着渲染 UI，先让数据变得可读、可抓、可维护。中间态也可直接成为回复文本内界面显示的一部分。

### 第六步：写回复代码

让它在每次回复时执行这套流程：

- 找到自己
- 判断自己是不是最新回复
- 等待初始化完成
- 抓取同级中间态
- 清空旧数据
- 投放新数据

### 第七步：回到平台做两种测试

你至少要测这两种情况：

1. 页面刚打开时，历史回复很多，是否只有最后一个回复真正更新了界面。
2. 新回复生成后，是否能正确读取这一条回复的中间态并刷新 UI。

只要这两种情况都稳定，说明这套基本架构已经跑通了。

## 各部分代码放置规则

- 界面代码放进功能栏，用 `<界面>` 这样的标记，正则替换成界面结构。

- CSS 代码也放进功能栏，用 `<美化>` 这样的标记，正则替换成样式内容。

- 初始化代码同样放进功能栏，用 `<初始>` 这样的标记，正则替换成初始化代码。

- 中间态正则与常规正则用法相同，放到“设置正则”里，并且按条保存。一条规则处理一类中间态。

- 回复代码可采用任意标记在每次回复中正则替换一次，例如要求每次回复包含 `回复开始` 或 `回复结束` 用于替换，或附着在某一条中间态正则如状态栏的替换内容里。
