第一部分：核心准则（含雷达式创作准则）
================================================================================

1. 目标导向原则
────────────────
本规范的核心目标是在 sexyai.ai 平台的限制下，实现稳定可靠的动态交互式
面板，同时最大化 Token 效率。所有技术决策都服务于这一目标。

本平台类似 Silly Tavern，平台本身已提供网页结构、消息流和基础交互。
开发任务不是重做整页网页，而是基于平台现有页面，按统一运行模型接入
任意定制UI，并把AI回复稳定投放到该UI中。

典型接入场景包括：
  - 对话界面
  - 状态栏
  - 地图界面
  - 日志面板
  - 卡片列表
  - 任务面板

2. 五段式交付原则
────────────────
  【第一部分：数据格式模板】  AI生成时需要填写的数据结构
  【第二部分：正则表达式配置】数据匹配与分段替换规则
  【第三部分：美化渲染模板】  分段式HTML+CSS代码
  【第四部分：AI生成规范】    确保AI每次生成时格式正确的强制性规则
  【第五部分：完整示例】      可直接测试的完整数据块

3. 性能优先原则
────────────────
采用 onerror 点火器模式进行运行时数据解析，所有JS逻辑集中在单个img
标签中执行，避免跨img状态传递问题。

4. 内容专注原则
────────────────
所有代码块内只包含纯粹的、可直接复制使用的内容。

5. 美化核心准则
────────────────
  - 可嵌入性：纯HTML片段，不含<html>/<head>/<body>
  - 样式内聚：CSS封装在<style>标签内
  - 绝对响应式：流体布局，自适应各种屏幕
  - 事件隔离：最外层容器包含 onclick="event.stopPropagation()"
  - 原子化CSS：独特前缀类名避免被主题覆盖

6. 数据继承原则
────────────────
  - 支持跨消息数据继承
  - 继承策略可配置
  - 长对话Token节省可达90%

7. DOM操作原则
────────────────
  推荐方式（最安全）：
    - document.createElement() + appendChild()
    - element.textContent / element.className

  允许方式（实测可用）：
    - innerHTML = ''（清空容器）
    - innerHTML = '...'（拼接赋值，需注意引号转义）
    - style.cssText = '...'（通过自定义函数间接设置）

  [推荐] 最佳实践：
    - 优先使用 createElement + appendChild 构建DOM
    - JS中只负责添加/切换类名和设置少量定位数值
    - 不在JS中拼接大段样式字符串

  语法要求：
    - const/let/var 均可使用（实测验证）
    - 箭头函数/传统function均可使用（实测验证）
    - .forEach()/.find()/.map()/.filter()等数组方法均可使用
    - 模板字符串``可使用
    - 解构赋值/展开运算符/可选链未经充分验证，不建议使用

  [推荐] 最佳实践：
    - 优先使用ES5写法以确保最大兼容性
    - 统一使用var、普通function、字符串拼接、显式判空
    - 但无需为避免ES6而扭曲代码逻辑

8. 局部刷新原则
────────────────
  - 禁止 location.reload()
  - 数据驱动局部DOM更新
  - 事件触发后 dispatchEvent({bubbles:true})

9. 单行化压缩原则
────────────────
  - onerror / onclick 属性中的JS必须压缩为严格的一行
  - 不允许包含任何 \n 或 \r 字符
  - 用分号 ; 替代所有换行作为语句分隔
  - 这是零级致命红线，违反则代码完全不执行
  - 可通过 data-s属性 或 <script type="text/plain"> 存储多行JS
    然后用单行eval语句触发执行，绕过此限制

10. 美化与功能分离原则
────────────────
  当美化代码（CSS+HTML骨架）与功能代码（JSON数据+JS逻辑+img触发器）
  合计超过20000字符时，必须将二者拆分为独立的链式规则：
    - 规则A（美化层）：<style>CSS + HTML骨架 + 链接标记
    - 规则B（功能层）：<script/json>数据 + <script/plain>JS + <img> + </div>
  分离后的维护策略：
    - 纯样式调整（颜色/字号/间距）→ 只改美化层
    - 纯逻辑调整（交互行为/数据处理）→ 只改功能层
    - 新增/删除HTML元素 → 两层都改

11. 完整代码输出原则
────────────────
  当用户要求修改代码时，必须输出该条规则完整的代码，
  禁止输出局部补丁或"找到这一行→替换为那一行"的片段指引。
  每次交付都是可直接部署的完整状态。

12. className压缩安全原则
────────────────
  在JS中拼接CSS类名时，必须确保压缩为单行后空格不会丢失。
  禁止使用三元运算符拼接空格的方式：
    ❌ d.className = 'z-cls' + (flag ? ' on' : '');
  推荐使用完整字符串赋值：
    ✅ d.className = flag ? 'z-cls on' : 'z-cls';
  或使用变量累加：
    ✅ var cn = 'z-cls'; if(flag) cn = cn + ' on'; d.className = cn;

13. 错误三次触发全面检查原则
────────────────
  当同一个问题修复三次仍未解决时，必须停止在当前假设方向上继续修补，
  转而对整个代码链进行全面系统性检查：
    - 检查所有链式规则的JSON括号层级匹配
    - 检查所有&继续N&标记的前后一致性
    - 检查script容器中是否存在\r字符
    - 检查replaceString是否超过20000字符上限
    - 增加try-catch诊断代码输出错误信息到页面可见区域

14. 文件拆分原则
────────────────
  整套实现在逻辑上固定拆成五类文件：
    1. 界面文件（DOM结构）
    2. CSS文件（样式）
    3. 初始化代码文件（启动逻辑）
    4. 回复代码文件（每次回复逻辑）
    5. 中间态正则列表文件（正则式+替换内容）

  必须拆分为独立文件维护，初始化代码和回复代码不混写。
  在平台链式规则体系中，每类文件对应一条或多条独立的正则规则。

15. 界面先行原则
────────────────
  先放界面结构，启动代码只初始化一次，每次回复代码按标准执行链运行。
  优先读取当前回复的同级中间态。

16. 规范优先级原则
────────────────
  执行时以本文件为准。如果其它文档、历史方案或旧模板与本文件冲突，
  应优先遵守本文件中的运行模型、拆分方式和触发规则。

17. 雷达式文本简洁性准则（创作规范）
────────────────
  - 删除冗余含义：若连续两个句子以不同措辞表达同一含义，仅保留其一。
  - 删除结构性旁白解释，禁止在描述中添加说明性语调。
  - 过渡应突然，不应被平滑处理。
  - 优先使用具体的、物理性的描述，而非比喻、类比或扩展隐喻。
  - 使用交替场景并置完成对比，而非编辑性对比陈述。
  - 禁止重复已出现的信息或句子，每个场景应引入特定叙事事件。
  - 时间客观流动，每个场景应引入特定叙事事件；对先前主题、动作或语言的任何重复或呼应均无价值。

平台独有特性：onerror → onload 自动转换
──────────────────────────────────────────
  部分角色卡包含一条"总体替换"规则：
    findRegex: /src\s=\s(["'])(?:x|)\1[\s\S]?onerror\s=/
    replaceString: src="data:image/gif;base64,R0lGOD..." onload=

  此规则将所有onerror转换为onload+有效gif。
  此规则是可选优化，不是必须的——部分角色卡有此规则，部分没有，
  两种情况都能正常工作。

[推荐] CSS预定义原则
──────────────────────────────────────────
  样式应集中在CSS文件（即<style>标签）中维护。
  JS中只负责：
    - 添加类名
    - 切换类名
    - 设置少量必要的定位数值
  不要在JS中拼接大段样式字符串，更不要依赖cssText。

[推荐] 避免重复ID原则
──────────────────────────────────────────
  如果界面方案中需要生成带标识的节点，不要使用会在多条消息中
  重复出现的固定ID。应优先：
    - 使用类名
    - 使用 data-* 属性
    - 使用时间戳或唯一后缀

[推荐] 正则替换内容控制原则
──────────────────────────────────────────
  中间态正则替换应尽量保持结构清晰、长度可控。
  不要在单条替换内容里塞入过长的大段代码或过度复杂的结构，
  以免出现截断、维护困难或平台兼容问题。

第三部分：运行模型与文件架构
================================================================================

3.1 五类文件架构
────────────────────
  整套实现在逻辑上固定拆成五类文件，每类文件在平台链式规则体系中
  对应一条或多条独立的正则规则：

  ┌───────┬──────────────────────┬────────────────────────────────────┐
  │ 编号  │ 文件类型             │ 职责                               │
  ├───────┼──────────────────────┼────────────────────────────────────┤
  │ 1     │ 界面文件             │ 长期存在的DOM结构，只负责显示内容  │
  │ 2     │ CSS文件              │ 界面样式，不负责数据逻辑           │
  │ 3     │ 初始化代码文件       │ 只在页面打开时运行一次             │
  │ 4     │ 回复代码文件         │ 每条回复触发时运行                 │
  │ 5     │ 中间态正则列表文件   │ 只负责"正则式+替换内容"            │
  └───────┴──────────────────────┴────────────────────────────────────┘

  关键规则：
    - 初始化代码和回复代码必须分开维护，不混写
    - 中间态正则列表只保留正则式和替换内容，不混入运行逻辑
    - 界面文件只负责DOM结构，不负责数据处理

3.2 对应平台正则规则的映射
────────────────────
  在平台的regex_scripts配置中，五类文件通常映射为：

  ┌──────────────────────┬──────────────────────────────────────┐
  │ 文件类型             │ 平台正则规则示例                     │
  ├──────────────────────┼──────────────────────────────────────┤
  │ CSS文件              │ findRegex:"<css>"                    │
  │                      │ replaceString:"<style>CSS代码</style>"│
  ├──────────────────────┼──────────────────────────────────────┤
  │ 初始化代码文件       │ findRegex:"<初始化>"                 │
  │                      │ replaceString:"<img onerror=...>"    │
  ├──────────────────────┼──────────────────────────────────────┤
  │ 界面文件             │ findRegex:"<界面>"                   │
  │                      │ replaceString:"全局界面HTML"         │
  ├──────────────────────┼──────────────────────────────────────┤
  │ 回复代码文件         │ findRegex:"<回复>"                   │
  │                      │ replaceString:"<img onerror=...>"    │
  ├──────────────────────┼──────────────────────────────────────┤
  │ 中间态正则列表       │ 各自独立的正则规则，如：             │
  │                      │ 状态栏正则、选项正则等               │
  └──────────────────────┴──────────────────────────────────────┘

3.3 statusbar字段的文件承载
────────────────────
  statusbar字段承载CSS文件、初始化代码文件和界面文件的触发标记：
    statusbar: "<css><初始化><界面>"

  这些标记在statusbar渲染时被对应的正则规则匹配替换，
  从而将CSS、初始化代码和界面结构注入到页面中。

3.4 beginning字段的文件承载
────────────────────
  beginning字段承载首条消息内容及回复级组件的触发标记。
  回复代码通常附加在选项正则的替换内容末尾：
    选项正则replaceString: "选项HTML代码<回复>"
    回复代码正则replaceString: "<img onerror=...>"

3.5 三层职责模型
────────────────────
  整套逻辑可理解为三层职责：

  A. 核心界面层
  ────────────────
    核心界面是页面中长期存在的DOM结构，负责显示内容。
    建议至少包含：
      - 外层容器：显示/隐藏、定位、层级、与原页面共存、
                  阻止事件冒泡干扰原页面交互
      - 展示区：渲染数据内容
      - 控制区：按钮、切换、输入等交互元素
      - 隐藏状态区：保存已迁移的数据，供翻页/切换/展开等逻辑读取

    外层容器事件隔离示例：
      <div class="ui-interface" onclick="event.stopPropagation()">

  B. 启动初始化层
  ────────────────
    启动代码只在页面打开时运行一次。
    适合放在启动代码里的内容：
      - 页面定位
      - 全局变量初始化
      - 全局控制对象初始化
      - 设置项读取与回填
      - 初始化完成标记
      - 缓存对象、定时器引用、运行锁初始化

    不应在启动代码里写某一条回复的具体业务数据。

  C. 每次回复运行层
  ────────────────
    回复代码在每次回复img触发时运行，按六步执行链组织。
    详见第六部分。


================================================================================
第四部分：视图分类与最新回复仲裁
================================================================================

4.1 界面架构分类
────────────────────
  为避免职责混乱，界面分为两类：

  ┌──────────────────────┬────────────────────────────────────────┐
  │ 类型                 │ 说明                                   │
  ├──────────────────────┼────────────────────────────────────────┤
  │ 全局单例视图         │ 页面级唯一，跨回复复用                 │
  │ (Global Singleton    │ 通常不随每条回复重复创建               │
  │  View)               │ 常见于：地图、侧边栏、弹窗、图鉴、    │
  │                      │ 全局面板、总控浮层                     │
  ├──────────────────────┼────────────────────────────────────────┤
  │ 回复级重复视图       │ 每条回复可有独立实例                   │
  │ (Reply-Scoped        │ 数据来源是当前回复同级中间态           │
  │  Repeated View)      │ 不应依赖"最新实例主执行段"才能显示     │
  │                      │ 常见于：状态栏块、选项块、日志块       │
  └──────────────────────┴────────────────────────────────────────┘

  两类可并存，但职责不能交叉。

4.2 最新回复仲裁机制
────────────────────
  为什么必须存在：
    页面打开时，历史回复对应的多个回复img会一起加载。
    这些img的加载顺序并不稳定，不能保证与回复时间顺序一致。
    如果不做最新仲裁，旧回复和新回复可能同时运行"全局单例视图"更新，
    导致：
      - 旧数据覆盖新数据
      - 面板重复刷新
      - 状态错乱
      - 调试困难

  仲裁规则：
    最新仲裁基于 .content.left 的DOM顺序，
    不基于图片加载顺序或loader下标。

  仲裁代码（可读格式，部署时须单行化）：
    function isLatestReplyBox() {
        var box = self ? self.closest('.content.left') : null;
        var replies = document.querySelectorAll('.content.left');
        var latestReply = replies[replies.length - 2];
        return !!(box && latestReply && box === latestReply);
    }

  说明：使用 replies.length - 2 而非 replies.length - 1，
  因为最后一个 .content.left 可能是正在生成中的回复容器，
  倒数第二个才是已完成的最新回复。

4.3 全局单例视图更新规则
────────────────────
  全局单例视图的更新（如侧边栏数据刷新、地图标记更新）
  只允许在最新回复仲裁通过后执行。

  回复级重复视图的渲染（如当前回复的状态栏块）
  不需要通过最新仲裁，每个回复实例可独立处理自身。


================================================================================
第五部分：触发机制
================================================================================

5.1 初始化img
────────────────────
  初始化代码由隐藏图片的onerror触发。

  要求：
    - 页面打开时运行一次
    - 只负责一次性初始化
    - 运行完成后应移除自身节点（this.remove()或removeChild）
    - 不处理某一条具体回复的数据
    - img标签放在所属容器内部

  示例结构：
    <img src="" style="display:none" onerror="初始化JS代码;this.remove()">

5.2 回复img
────────────────────
  回复代码由回复区域中的回复img触发。

  要求：
    - 每条回复都拥有唯一自己的回复img
    - 历史回复在页面打开时可能同时触发
    - 新生成回复会在该条回复完成生成且中间态替换完成后触发
    - 回复代码按"六步执行链"组织（按需省略可选步骤）
    - img只负责触发，不作为新旧回复仲裁依据
    - img标签放在所属容器内部

5.3 新生成回复的执行理解
────────────────────
  新生成回复的回复img会在以下条件满足后才触发：
    1. 该条回复已经完成生成
    2. 平台已经完成该条回复的中间态正则替换
    3. 回复img开始加载并触发回复代码

  因此，新生成回复一旦触发，通常已经是当前最新回复，
  并且同级中间态应已可读取。

5.4 运行触发架构总览
────────────────────
  运行触发以多个img为基础：一个初始化img，多个回复img。

  ┌──────────────┬────────────────┬───────────────────────┐
  │ img类型      │ 触发时机       │ 执行次数              │
  ├──────────────┼────────────────┼───────────────────────┤
  │ 初始化img    │ 页面打开       │ 仅一次                │
  │ 回复img      │ 每条回复完成   │ 每条回复各一次        │
  │              │ 页面打开时历史 │ 历史回复可能同时触发  │
  └──────────────┴────────────────┴───────────────────────┘
================================================================================
第六部分：执行链规范
================================================================================

6.1 六步执行链概述
────────────────────
  回复代码在每次回复img触发时运行。
  固定执行链采用六步结构（部分步骤可按需省略）：

  ┌───────┬──────────────────────────────────┬──────────┐
  │ 阶段  │ 名称                             │ 是否必须 │
  ├───────┼──────────────────────────────────┼──────────┤
  │ 1     │ 固定引用段（Self Capture）        │ ✅ 必须  │
  │ 2     │ 初始化依赖协商段                 │ 可选     │
  │       │ (Initialization Dependency Gate) │          │
  │ 3     │ 当前回复自处理段                 │ 可选     │
  │       │ (Reply-Scoped Process)           │          │
  │ 4     │ 最新回复仲裁段                   │ 可选     │
  │       │ (Latest-Reply Arbitration)       │          │
  │ 5     │ 最新回复处理段                   │ 可选     │
  │       │ (Latest-Only Main Stage)         │          │
  │ 6     │ 结束自删段（Final Self Cleanup）  │ ✅ 必须  │
  └───────┴──────────────────────────────────┴──────────┘

6.2 Stage 1：固定引用段（Self Capture）
────────────────────
  回复代码开头先固定 self = this。
  确保后续异步回调中仍能引用到当前img元素。

  代码：
    var self = this;

6.3 Stage 2：初始化依赖协商段（可选）
────────────────────
  仅当后续逻辑依赖初始化能力（如全局控制对象、页面定位完成）时才等待。
  采用"依赖驱动"模式，不是无条件等待。

  代码（可读格式，部署时须单行化）：
    function waitReady(next) {
        if (!window.uiReady || !window.uiApp) {
            setTimeout(function() {
                waitReady(next);
            }, 50);
            return;
        }
        next();
    }

6.4 Stage 3：当前回复自处理段（可选）
────────────────────
  每个回复实例可先处理自身所属的回复容器，不依赖"最新实例"判定。
  适用于回复级重复视图（状态栏块、选项块、日志块）。
  优先抓同级元素，不优先全局搜索。

  代码（可读格式，部署时须单行化）：
    function runScoped() {
        var box = self ? self.closest('.content.left') : null;
        if (!box) return;
        // 处理当前回复同级中间态
    }

6.5 Stage 4：最新回复仲裁段（可选）
────────────────────
  只有在需要"最新实例处理"（全局单例视图更新）时才执行本段。
  当前回复容器不是最新回复则立即自删退出。

  代码（可读格式，部署时须单行化）：
    function isLatestReplyBox() {
        var box = self ? self.closest('.content.left') : null;
        var replies = document.querySelectorAll('.content.left');
        var latestReply = replies[replies.length - 2];
        return !!(box && latestReply && box === latestReply);
    }

  说明：
    - 仲裁基于 .content.left 的DOM顺序
    - 不基于图片加载顺序或loader下标
    - 使用 length - 2 是因为最后一个可能是生成中的容器

6.6 Stage 5：最新回复处理段（可选）
────────────────────
  仅在Stage 4判定通过后执行，用于全局单例视图更新。
  如侧边栏数据刷新、地图标记更新、全局面板内容替换等。

  代码（可读格式，部署时须单行化）：
    function runLatestOnly() {
        // 全局单例视图更新
    }

6.7 Stage 6：结束自删段（Final Self Cleanup）
────────────────────
  执行完当前脚本涉及的所有逻辑后，直接删除自身img节点。
  初始化img和回复img在执行完成后都应自删。

  代码：
    function removeSelf() {
        if (self && self.parentNode) self.parentNode.removeChild(self);
    }

6.8 组合方式（按需取舍）
────────────────────
  组合A：只做"每条回复各自处理"
    使用 Stage 1 + Stage 3 + Stage 6

  组合B：需要"最新回复处理"
    使用 Stage 1 +（可选 Stage 2）+（可选 Stage 3）
         + Stage 4 + Stage 5 + Stage 6

6.9 统一模板（可读格式，部署时须单行化）
────────────────────
 var self = this;

function removeSelf() {
    if (self && self.parentNode) self.parentNode.removeChild(self);
}

function runScoped() {
    var box = self ? self.closest('.content.left') : null;
    if (!box) return;
    // 处理当前回复同级中间态
}

function isLatestReplyBox() {
    var box = self ? self.closest('.content.left') : null;
    var replies = document.querySelectorAll('.content.left');
    var latestReply = replies[replies.length - 2];
    return !!(box && latestReply && box === latestReply);
}

function runLatestOnly() {
    // 全局单例视图更新
}

function main() {
    runScoped();
    if (isLatestReplyBox()) {
        runLatestOnly();
    }
    removeSelf();
}

function waitReady(next) {
    var maxTries = 120;
    var tries = 0;
    (function checkReady() {
        if (!window.uiReady || !window.uiApp) {
            tries = tries + 1;
            if (tries >= maxTries) {
                removeSelf();
                return;
            }
            setTimeout(checkReady, 50);
            return;
        }
        next();
    })();
}

第七部分：脚本生存与逻辑执行规范
================================================================================

7.1 JS执行策略（6种已验证模式）
────────────────────────────────

  模式1：onerror + (function(img){...})(this)
    特点：img参数指向img元素，可用closest()作用域查找
    适用：需要在特定容器内查找元素时

  模式2：onerror + (function(){...})()
    特点：无参数，使用document.getElementById()全局查找
    适用：使用唯一ID定位元素时

  模式3：onerror + 直接赋值window全局对象
    代码：onerror="window.MyObj={method1:function(){...},...};"
    特点：将整个应用逻辑挂载到window全局对象
    适用：需要跨元素调用方法时

  模式4：eval(document.getElementById('xx').dataset.s)
    代码：onclick="eval(document.getElementById('FUNC_ID').dataset.s)"
    特点：JS代码存储在隐藏<p>的data-s属性中，按需eval执行
    适用：需要复用同一段JS逻辑的多个按钮
    优势：data-s中的JS可以包含换行！绕过onerror单行限制

  模式5：eval(document.getElementById('xx').textContent)
    代码：<script type="text/plain" id="code">多行JS</script>
         <img onerror="eval(document.getElementById('code').textContent)">
    特点：JS代码存储在<script type="text/plain">中，可多行
    适用：极长的JS逻辑（如抽卡系统、属性计算）
    注意：<script type="text/plain">中绝对禁止\r字符

  模式6：内联onclick直接编写JS
    代码：onclick="var a=1;dosomething()"
    特点：最简单直接，代码必须在一行内
    适用：简短的交互逻辑

7.2 初始化代码应包含的内容
────────────────────────────────

  A. 全局变量
    至少初始化这些类型的内容：
      - 初始化完成标记
      - 全局控制对象
      - 当前状态变量
      - 可选的设置项默认值
      - 可选的定时器或缓存引用

    示例：
      window.uiReady = false;
      window.uiApp = {};
      window.__currentIndex = 0;
      window.__panelOpen = false;

  B. 全局控制对象
    推荐统一挂到 window.uiApp：
      window.uiApp = {
          open: function(){},
          close: function(){},
          render: function(){},
          reset: function(){},
          ensureReady: function(){}
      };

  C. 页面定位或布局同步
    界面嵌在原网页中，初始化时应优先处理页面定位，
    避免遮挡顶部栏和底部栏。
    界面层级通常控制在100左右。

    推荐定位代码（可读格式，部署时须单行化）：
      var uiRoot = document.querySelector('.ui-interface');
      var bottomBar = document.querySelector('.chat-bottom');
      var topBar = document.querySelector('.topTabbar');

      if (uiRoot && bottomBar && topBar) {
          var bottomHeight = bottomBar.offsetHeight;
          var visibleHeight = window.innerHeight
              - topBar.offsetHeight - bottomHeight;

          uiRoot.style.bottom = bottomHeight + 'px';
          uiRoot.style.height = visibleHeight + 'px';
          uiRoot.style.zIndex = 100;
      }

  D. 初始化完成标记
    初始化逻辑末尾必须设置：
      window.uiReady = true;

  E. 启动代码触发节点用完后移除
    初始化完成后应移除初始化img节点：
      this.remove();

7.3 输入框选择器
────────────────────
  推荐方案（覆盖率最高）：
    var c = document.querySelector('.chatMsgTextarea');
    if(c){ var i = c.querySelector('textarea') || c; ... }

  完整备选列表：
    var S = [
      '.chat .chat-bottom .uni-textarea .chat-input-scope textarea',
      '#send_textarea textarea',
      '#send_textarea',
      'textarea',
      '[contenteditable=true]'
    ];

7.4 输入同步
────────────────────
  写入输入框后必须触发：
    i.dispatchEvent(new Event('input', {bubbles:true}))

  不调用则平台不会识别输入变化。

第八部分：UI / CSS 布局规范
================================================================================

8.1 position: fixed 组件
  所有侧边栏/游玩助手/特化库/悬浮球均使用 position:fixed
  z-index 范围：999 到 999999999

  [推荐]：界面层级通常控制在100左右，避免压过过多
  原页面元素，也避免被原页面普通内容盖住。全局单例视图（如侧边栏）
  可使用更高z-index。

8.2 writing-mode: vertical-rl
  侧边栏触发按钮普遍使用竖排文字

8.3 backdrop-filter: blur()
  模态窗口背景模糊效果可正常工作

8.4 CSS动画 @keyframes
  平台完全支持CSS动画，包括渐变/脉冲/发光/扫描线动画

8.5 @import url() 外部资源
  可正常加载Google Fonts和外部CSS
  也可加载外部图片、视频、音频（无严格CSP限制）

8.6 CSS自定义属性（变量）
  完全正常工作，推荐用于实现可维护的主题系统
  示例：--bg:#0d1117; --accent:#58a6ff;

8.7 全局美化
  平台的几乎所有UI元素都可以通过CSS !important覆盖
  图标通过filter属性改色（brightness/invert/sepia/hue-rotate组合）
  详见第二十三部分完整选择器清单

8.8 响应式设计（@media断点）
  平台完全支持@media查询，推荐以下断点适配全端：
  ┌──────────────────┬──────────┬───────────────────────────────┐
  │ 设备类型         │ 断点     │ 推荐max-width               │
  ├──────────────────┼──────────┼───────────────────────────────┤
  │ 手机             │ <768px   │ 100%（最大520px）            │
  │ 平板             │ ≥768px   │ 600-640px                    │
  │ 电脑             │ ≥1200px  │ 680-720px + 圆角 + 阴影     │
  │ 大屏             │ ≥1600px  │ 760-800px                    │
  └──────────────────┴──────────┴───────────────────────────────┘

  电脑端居中显示的关键CSS：
    .content.left .z-xxx-box {
      margin-left: auto !important;
      margin-right: auto !important;
      float: none !important;
    }
  此规则覆盖平台AI气泡容器的默认对齐方式。

  电脑端增加"卡片感"的推荐CSS：
    border-left: 1px solid var(--border);
    border-right: 1px solid var(--border);
    border-radius: 16px;
    box-shadow: 0 0 60px rgba(0,0,0,0.4);

8.9 界面和样式基本要求
────────────────────
  - 界面以可嵌入为前提：输出应是能嵌入平台现有页面的HTML片段，
    不要生成整页HTML文档结构，不要包含<html>/<head>/<body>
  - 样式内聚：样式应集中写在CSS文件（<style>标签）中，
    由界面结构通过类名引用
  - 响应式和共存：界面应默认考虑不同屏幕尺寸，
    并与平台原有顶部栏、底部栏、消息区域共存
  - 可读性优先：颜色、层级、字号和间距应优先保证可读性，
    不要为了装饰牺牲信息识别

================================================================================
第十部分：中间态标准链路
================================================================================

10.1 中间态概念
────────────────────
  中间态是"原始AI文本"到"界面渲染"之间的结构化中间产物。
  设计回复投放方案时，应优先引入中间态，避免直接在大段原始文本中
  完成复杂界面渲染。

  中间态的职责是"把原始文本变成结构化输入"，
  不是直接承担整套交互逻辑。

10.2 标准链路
────────────────────
  完整的数据流经历五个阶段：

  ┌─────┬──────────────────────┬──────────────────────────┐
  │ 步骤│ 阶段                 │ 说明                     │
  ├─────┼──────────────────────┼──────────────────────────┤
  │ 1   │ 原始AI文本           │ AI按约定格式生成文本     │
  │ 2   │ 正则替换成结构化节点 │ 平台正则将文本转为       │
  │     │                      │ 隐藏div/data-属性/子节点 │

  └─────┴──────────────────────┴──────────────────────────┘

10.3 推荐中间态形式
────────────────────
  - 隐藏div：<div style="display:none" class="z-data">数据</div>
  - data-*属性：<div data-name="值" data-hp="值"></div>
  - 子节点文本：<div class="z-item"><span>名称</span></div>
  - 多个同级节点：同一容器内排列多个结构化数据节点

10.4 中间态正则列表要求
────────────────────
  中间态正则列表必须独立维护。
  该文件中只保留两类内容：
    - 正则式
    - 替换内容

  不要在中间态正则列表中混入：
    - 初始化逻辑
    - 回复逻辑
    - 大段运行说明
    - 与页面定位无关的控制代码

  目标是让中间态列表只负责把原始回复转换成结构化节点。

11.2 继承代码（可读格式，部署时须单行化）
────────────────────
  var findData = function(selector, currentOnly) {
    var dataBox = box.querySelector('.z-status-data');
    if (dataBox) {
      var found = dataBox.querySelector(selector);
      if (found) return found;
    }
    if (currentOnly) return null;
    var allBoxes = document.querySelectorAll('.z-status-box');
    var idx = -1;
    for (var i = 0; i < allBoxes.length; i++) {
      if (allBoxes[i] === box) { idx = i; break; }
    }
    for (var j = idx - 1; j >= 0; j--) {
      var prevData = allBoxes[j].querySelector('.z-status-data');
      if (prevData) {
        var found = prevData.querySelector(selector);
        if (found) return found;
      }
    }
    return null;
  };

11.3 继承策略与Token节省
────────────────────
  通过数据继承机制，AI在后续回复中只需提供变化的字段，
  未变化字段从历史回复中向前查找继承。
  长对话场景下Token节省可达90%。

12.1 链式规则模式
────────────────────────────────
  当单条replaceString超过约20000字符时，必须拆分为多条链式规则。

  方式A：&中文标记& 模式（最常用）
    规则1: findRegex: "触发文本"    → replaceString: "CSS+HTML+&继续2&"
    规则2: findRegex: "&继续2&"     → replaceString: "更多HTML+&继续3&"
    规则末: findRegex: "&继续末&"   → replaceString: "img onerror+闭合div"

  方式B：<!-- 英文标记 --> 模式
    规则1: findRegex: "<status>"    → replaceString: "CSS+<!-- Z_CONTENT -->"
    规则2: findRegex: "<!-- Z_CONTENT -->" → replaceString: "HTML+<!-- Z_SCRIPT -->"
    规则3: findRegex: "<!-- Z_SCRIPT -->"  → replaceString: "img onerror+</div>"

  方式C：HTML占位符模式
    规则1: findRegex: "触发文本"    → replaceString: "含<div id='SLOT'></div>"
    规则2: findRegex: "/<div id=\"SLOT\"><\\/div>/"
           → replaceString: "填充内容"

  关键规则：
    - 标记在findRegex和前一条的replaceString中必须完全一致
    - 每条replaceString控制在20000字符以内
    - CSS和HTML部分可以包含换行
    - 最后一条规则包含img onerror（JS必须单行）
    - img标签必须在最外层容器的闭合</div>之前

12.2 JS代码格式要求
────────────────────────────────
  本文档中JS代码以多行格式展示仅为可读性。
  实际部署时：
    ⚠️ onerror="..." 双引号内的全部JS必须压缩为一行
    ⚠️ 不允许存在任何 \n 或 \r 字符
    ⚠️ 用分号 ; 替代所有换行

  绕过方案：
    将长JS存储在data-s属性或<script type="text/plain">中
    onerror仅需一行eval语句来触发执行
    注意：<script type="text/plain">中绝对禁止\r字符

12.3 多img执行顺序
────────────────────────────────
  同一容器内的多个img按DOM顺序依次执行。
  可利用window全局变量在多个img之间传递数据。
  前一个img挂载到window的数据，后一个img可以读取。

12.4 JSON链式数据完整性校验
────────────────────────────────
  当JSON数据通过多条链式规则传递时（如大量标签数据分多条规则），
  每条规则的JSON片段必须在链标记衔接处保持括号和逗号的严格匹配。

  校验方法：
    - 每条规则结尾的闭合括号只关闭当前应关闭的层级
    - 下条规则开头从上条停止的层级正确接续
    - 最后一条规则的结尾精确关闭所有剩余嵌套层级

  常见错误：
    ❌ 规则N结尾多了一组]}，提前关闭了上层数组/对象
    ❌ 规则N+1开头缺少逗号，导致JSON语法错误

  诊断方法：
    在img onerror中用try-catch包裹JSON.parse()，
    catch块将错误信息输出到页面可见区域：
    try{window._zxpD=JSON.parse(el.textContent)}
    catch(e){pvEl.textContent='JSON ERROR: '+e.message;return}

  错误信息解读：
    "unexpected non-whitespace character after JSON data at column N"
      → JSON根对象在column N处被提前闭合，后面还有数据
      → 某条规则结尾多了闭合括号

    "expected ',' or ']' in array at column N"
      → 两条规则拼接处缺少逗号
      → 检查&继续N&标记前后的语法

第十三部分：结构与交互规范

13.1 事件隔离
────────────────────
  最外层容器必须包含 onclick="event.stopPropagation()"。
  所有模态窗口内容层、侧边栏面板、下拉菜单均需同样处理。
  不加则UI组件的点击事件会冒泡触发平台的对话气泡行为。

13.2 模态窗口
────────────────────
  基本结构：
    背景层：position:fixed; inset:0; background:rgba(0,0,0,0.8);
           onclick关闭自身
    内容层：onclick="event.stopPropagation()"
           阻止点击内容时关闭窗口
    关闭按钮：onclick移除active/open类名

  打开方式：classList.add('active') 或 style.display='flex'
  关闭方式：classList.remove('active') 或 style.display='none'

13.3 折叠交互
────────────────────
  方案A（JS）：onclick切换classList，配合max-height:0→500px过渡
  方案B（CSS纯交互）：<details><summary>标签
  方案C（CSS Radio）：radio input + :checked选择器实现互斥切换

13.4 标签页切换
────────────────────
  方案A（JS）：onclick遍历所有tab移除active，当前tab添加active，
              同时切换对应内容面板的display
  方案B（CSS纯交互）：radio input + :checked + 相邻兄弟选择器
              无需任何JS，完全由CSS控制显示隐藏

13.5 数据回显
────────────────────
  删除操作使用 split(o).join('') 或 replace 兼容各种空格/换行情况。
  预览区（dialog）与发送区同步逻辑必须一致。

14.1 localStorage
────────────────────
  保存：localStorage.setItem('key_' + timestamp, value)
  读取：var v = localStorage.getItem('key_' + timestamp)
  删除：localStorage.removeItem('key_' + timestamp)

  使用时间戳后缀确保多个角色卡实例不冲突。
  适用于跨页面保存角色创建选择、游戏进度等。

14.2 延迟回显
────────────────────
  读取localStorage或初始化状态时封装setTimeout(fn, 500)，
  在回调内查找挂载点并渲染状态，以兼容渲染延迟。

14.3 window全局变量
────────────────────
  可用于在同一页面的多个img标签之间传递数据。
  window.APP_DATA = {...}; 在第一个img中设置，
  后续img中读取window.APP_DATA。

  [推荐]推荐统一挂载到window.uiApp全局控制对象，
  避免命名空间污染。

14.4 dataset属性
────────────────────
  element.dataset.key = value; 用于元素级数据存储。
  适用于按钮绑定命令文本、存储状态标志等。

第十六部分：实战压缩模板
================================================================================

  所有onclick/onerror模板必须为单行。

  输入框注入模板：
    onclick="var t=document.querySelector('.chatMsgTextarea');
    if(t){var i=t.querySelector('textarea')||t;
    i.value='注入文本';
    i.dispatchEvent(new Event('input',{bubbles:true}))}"

  输入框追加模板：
    onclick="var t=document.querySelector('.chatMsgTextarea');
    if(t){var i=t.querySelector('textarea')||t;
    i.value+=this.getAttribute('data-t');
    i.dispatchEvent(new Event('input',{bubbles:true}))}"

  输入框清空模板：
    onclick="var t=document.querySelector('.chatMsgTextarea');
    if(t){var i=t.querySelector('textarea')||t;
    i.value='';i.dispatchEvent(new Event('input',{bubbles:true}))}"

  注意：以上模板展示为多行仅为可读性，实际使用时必须压缩为一行。

  className压缩安全模板：
    ❌ 危险写法（压缩后空格可能丢失）：
       d.className = 'z-cls' + (flag ? ' on' : '');

    ✅ 安全写法A（完整字符串赋值）：
       d.className = flag ? 'z-cls on' : 'z-cls';

    ✅ 安全写法B（变量累加）：
       var cn = 'z-cls'; if(flag) cn = cn + ' on'; d.className = cn;

  中文字符压缩安全模板：
    在需要单行压缩的JS代码中，中文字符串推荐使用Unicode转义：
      '【姓名】' → '\u3010\u59D3\u540D\u3011'
      '、' → '\u3001'
    这样确保压缩为单行时不会产生编码问题。

18.2 JSON诊断模板
────────────────────
  在img onerror中增加try-catch包裹JSON.parse：

  错误信息速查：
    "unexpected non-whitespace character after JSON data at column N"
      → 某条规则结尾多了闭合括号，JSON根对象被提前关闭
    "expected ',' or ']' at column N"
      → 两条规则拼接处缺少逗号
    "unterminated string at column N"
      → JSON字符串中的引号未正确转义

18.3 错误三次触发全面检查流程
────────────────────
  当同一个问题修复三次仍未解决时，执行以下系统性检查：
    1. 在img onerror中增加try-catch诊断代码
    2. 确认JSON.parse是否成功（如失败则问题在数据链）
    3. 确认eval是否成功（如失败则问题在JS代码）
    4. 逐条检查所有链式规则的括号层级
    5. 检查所有script容器中是否存在\r字符
    6. 检查所有replaceString是否超过20000字符
    7. 对比已知可运行版本与当前版本的差异

20.12 六步执行链模板
────────────────────
  以下为完整模板（可读格式，部署时须单行化）：

  var self = this;
  function removeSelf() {
      if (self && self.parentNode) self.parentNode.removeChild(self);
  }
  function runScoped() {
      var box = self ? self.closest('.content.left') : null;
      if (!box) return;
      // 处理当前回复同级中间态
  }
  function isLatestReplyBox() {
      var box = self ? self.closest('.content.left') : null;
      var replies = document.querySelectorAll('.content.left');
      var latestReply = replies[replies.length - 2];
      return !!(box && latestReply && box === latestReply);
  }
  function runLatestOnly() {
      // 全局单例视图更新
  }
  function main() {
      runScoped();
      if (window.__needLatestOnly) {
          if (!isLatestReplyBox()) { removeSelf(); return; }
          runLatestOnly();
      }
      removeSelf();
  }
  if (window.__needInitReady) {
      (function waitReady() {
          if (!window.uiReady || !window.uiApp) {
              setTimeout(waitReady, 50); return;
          }
          main();
      })();
  } else { main(); }

第二十一部分：正则匹配模式分类
================================================================================

21.1 纯文本匹配（最简单）
────────────────────
  findRegex值不包含/包裹，直接匹配beginning或statusbar中的文本。

  示例：
    findRegex: "开局美化（只在第一句话使用）"
    findRegex: "&全局美化&"
    findRegex: "【悬浮球】"
    findRegex: "&继续7&"
    findRegex: "<css>"
    findRegex: "<初始化>"
    findRegex: "<界面>"
    findRegex: "<回复>"

  触发标记命名惯例：
    &标记名& — 最常用
    【标记名】— 可视化更强
    &&标记名 — 双&区分优先级
    ◎标记名 — 特殊符号
    <标记名> —推荐（如<css>/<初始化>/<界面>/<回复>）
    标记名（后续不需要生成）— 含使用说明

21.2 正则表达式匹配（复杂数据提取）
────────────────────
  findRegex值以/包裹，使用正则表达式进行模式匹配。
  通过捕获组$1 $2 $3...提取数据填入replaceString。

  常用修饰符：
    /s — dotAll模式，使.能匹配换行符
    /g — 全局匹配，匹配所有出现位置

  常用模式：
    定界符模式：/START▶([\s\S]*?)◀END/
    标签模式：/<tag>(.*?)<\/tag>/s
    分隔符模式：/【开始】\s*(.*?)\s*\|\s*(.*?)\s*【结束】/s
    否定字符组：/[^◀]+/ 在定界符之间提取
    多捕获组：最多可达23个捕获组

21.3 HTML占位符匹配
────────────────────
  匹配由链式规则前一条插入的HTML标记。

  示例：
    findRegex: "/<!-- DGS_S3 -->/"
    findRegex: "/<div id=\"SLOT\"><\\/div>/"

21.4 总体替换规则（可选优化）
────────────────────
  将所有onerror转换为onload+有效gif图片。

  findRegex: /src\s=\s(["'])(?:x|)\1[\s\S]?onerror\s=/
  replaceString: src="data:image/gif;base64,R0lGOD..." onload=

  此规则是可选的，不是所有角色卡都需要。

21.5 中间态正则的设计原则
────────────────────
  中间态正则列表必须独立维护，只保留：
    - 正则式
    - 替换内容

  不要在正则列表中混入运行逻辑或大段说明。
  目标是让正则列表只负责把原始回复转换成结构化节点。

  推荐中间态替换结果：
    - 隐藏div：<div style="display:none" class="z-data">数据</div>
    - data-*属性节点：<div data-name="值" data-hp="值"></div>
    - 子节点文本结构：<div class="z-item"><span>名称</span></div>

第二十二部分：beginning与statusbar设计规范
================================================================================

22.1 beginning字段
────────────────────
  beginning是用户开始新对话时显示的第一条消息内容。
  其中的文本会被正则规则扫描和替换。

  设计模式：
    A. 正文 + 触发标记：
       正文剧情内容
       开局美化（只在第一句话使用）

    B. 触发标记 + 隐藏示例：
       &开局
       <!-- 给AI看的状态栏格式示例 -->

    C. 完整替换文本：
       整段beginning被正则完整匹配并替换为HTML

    D. 多标记组合：
       【悬浮球】【图鉴】【地图】【全局美化】

  [推荐]beginning承载首条消息内容及回复级组件
  的触发标记。回复代码通常附加在选项正则的替换内容末尾。

22.2 statusbar字段
────────────────────
  statusbar中的内容显示在平台界面的特定位置（非对话区），
  其中的标记文本同样会被正则匹配替换。

  适合放置：
    - 全局UI组件（侧边栏/工具栏）
    - 全局CSS美化样式
    - 初始化代码触发器
    - 全局界面结构
    - 滚动挂件
    - 自动判定系统

  示例（传统格式）：
    statusbar: "&全局美化&\n&上下滚动&\n&游玩助手&"

  [推荐] 标准格式：
    statusbar: "<css><初始化><界面>"

  此格式将CSS文件、初始化代码文件和界面文件的触发标记
  集中放在statusbar中，由对应的正则规则匹配替换后注入页面。

22.3 设计原则
────────────────────
  - statusbar更适合"全局UI组件"和"一次性初始化"
  - beginning更适合"首条消息内容"和"回复级组件触发"
  - 二者的标记不会互相干扰
  - 可以使用相同的命名规则

22.4 pageDepth字段
────────────────────
  值为2或null。使用pageDepth=2的配置倾向于有更复杂的UI。
  不影响正则功能本身。

    var e=document.querySelectorAll('.uni-scroll-view');
    var scroller=e.length>1?e[1]:e[0];

  滚动到顶：scroller.scrollTo({top:0, behavior:'smooth'})
  滚动到底：scroller.scrollTo({top:scroller.scrollHeight, behavior:'smooth'})

24.7 折叠/FAQ组件
────────────────────
  方案A：onclick切换classList('expanded')
         配合max-height:0→500px过渡动画
  方案B：<details><summary>（最简单但样式受限）

24.8 核心界面结构模板
────────────────────
  全局单例视图推荐结构：

  <div class="ui-interface" onclick="event.stopPropagation()">
    <!-- 展示区 -->
    <div class="ui-display">
      <!-- 数据渲染区域 -->
    </div>
    <!-- 控制区 -->
    <div class="ui-controls">
      <!-- 按钮、切换、输入等交互元素 -->
    </div>
    <!-- 隐藏状态区 -->
    <div class="ui-state" style="display:none">
      <!-- 保存已迁移的数据，供翻页/切换/展开等逻辑读取 -->
    </div>
    <!-- 初始化img放在容器内部 -->
    <img src="" style="display:none" onerror="初始化JS;this.remove()">
  </div>

  关键要点：
    - 外层容器负责显示/隐藏、定位、层级、事件隔离
    - img标签必须在闭合</div>之前
    - 初始化img执行完成后自删


================================================================================
第二十五部分：美化与功能分离架构规范
================================================================================

25.1 分离时机
────────────────────
  当美化代码（<style>CSS + HTML骨架）与功能代码（<script>JSON数据 +
  <script>JS逻辑 + <img>触发器 + </div>闭合）的总字符数超过20000时，
  必须将二者拆分为独立的链式规则。

25.2 分离结构
────────────────────
  规则1（美化层）：
    <div class="z-xxx-box" onclick="event.stopPropagation()">
    <style>全部CSS</style>
    全部HTML骨架（含所有id占位容器、输入框、按钮）
    &继续功能&

  规则2+（功能层，可进一步拆分为多条）：
    <script type="application/json" id="xxxD">JSON数据</script>
    <script type="text/plain" id="xxxL">JS逻辑</script>
    <img src="x" onerror="...">
    </div>

  关键约束：
    - 美化层的HTML骨架中所有id必须与功能层JS的querySelector完全一致
    - 功能层的img标签必须在最外层容器的闭合</div>之前
    - 链接标记（如&继续功能&）在两层的replaceString中必须完全一致

25.3 维护策略
────────────────────
  ┌─────────────────────────┬───────────────────┬───────────┐
  │ 修改类型                │ 需要改哪层         │ 示例       │
  ├─────────────────────────┼───────────────────┼───────────┤
  │ 纯样式（颜色/字号/间距）│ ✅ 只改美化层     │ 改配色方案 │
  │ 纯逻辑（交互/数据处理） │ ✅ 只改功能层     │ 改选择上限 │
  │ 纯数据（增删标签/描述） │ ✅ 只改功能层     │ 加新标签   │
  │ 新增模块（HTML+JS）     │ ⚠️ 两层都改      │ 加搜索框   │
  │ 删除模块（移除HTML+JS） │ ⚠️ 两层都改      │ 去掉组件   │
  │ 重命名元素id            │ ⚠️ 两层都改      │ 改id名称   │
  └─────────────────────────┴───────────────────┴───────────┘

25.4 JSON数据链拆分规范
────────────────────
  当JSON数据量过大需要拆分为多条链式规则时：
    - 每条规则结尾必须停在JSON语法安全点
      （如数组元素的逗号后、对象的闭合}后跟逗号）
    - 下条规则必须从语法安全点正确接续
    - 最后一条规则必须精确关闭所有剩余的嵌套层级
    - 每条规则控制在20000字符以内

  安全点示例：
    ✅ ...最后一个opt对象}]},&继续N&
    ✅ ...最后一个l2对象}]},&继续N&
    ❌ ...最后一个opt对象}]}]},&继续N&（多关闭了一层）

25.5 功能层JSON解析容错
────────────────────
  推荐在img onerror中使用try-catch包裹JSON.parse：
    try{window._D=JSON.parse(el.textContent)}
    catch(e){pvEl.textContent='JSON ERROR: '+e.message;return}

  这样当JSON数据链存在语法错误时，错误信息直接显示在页面上，
  帮助快速定位问题所在的链式规则环节。

25.6 五文件架构与美化/功能分离的关系
────────────────────
  五文件架构（界面/CSS/初始化/回复代码/中间态正则）
  与美化/功能分离是互补的两套组织维度：

  ┌──────────────────────┬──────────────────────────────────────┐
  │ 五文件架构维度       │ 美化/功能分离维度                    │
  ├──────────────────────┼──────────────────────────────────────┤
  │ 界面文件 + CSS文件   │ 对应美化层                           │
  │ 初始化代码文件       │ 对应功能层（一次性初始化部分）       │
  │ 回复代码文件         │ 对应功能层（每次回复运行部分）       │
  │ 中间态正则列表文件   │ 独立维护，不属于美化层或功能层       │
  └──────────────────────┴──────────────────────────────────────┘

  两套维度可叠加使用：
    - 按五文件架构确定逻辑职责分离
    - 当单条规则超长时按美化/功能分离拆分链式规则

附录：快速参考

A. 换行规则速查
────────────────────
  onerror中的JS           → ❌ 绝对禁止换行
  onclick中的JS            → ❌ 绝对禁止换行
  CSS / HTML               → ✅ 允许换行
  <script>数据/代码容器    → ✅ 允许换行（禁止\r，只允许\n）
  data-s / data-t 属性     → ✅ 允许换行

B. JS执行模式速查（6种）
────────────────────
  1. onerror + (function(img){...})(this)
  2. onerror + (function(){...})()
  3. onerror + window.OBJ = {...}
  4. onclick + eval(el.dataset.s)
  5. onerror + eval(el.textContent)  ← 注意禁止\r字符
  6. onclick + 直接内联JS

C. 数据存储速查
────────────────────
  <script type="application/json">  → JSON数据
  <script type="text/plain">        → 文本或JS代码（禁止\r）
  <p style="display:none" data-s>   → JS函数存储
  <div style="display:none">        → 原始文本数据/中间态
  localStorage                      → 跨页面持久化
  dataset / data-属性                → 元素级数据

D. 链式规则速查
────────────────────
  &中文标记& → 最常用
  <!-- 英文标记 --> → 备选
  {特殊标记} → 备选
  <标记名> →推荐
  规则数上限实测可达33条
  每条replaceString控制在20000字符以内

E. 正则匹配速查
────────────────────
  纯文本：直接匹配beginning/statusbar中的文本
  正则表达式：/pattern/s 用于提取AI输出的状态栏数据
  HTML占位：匹配链式规则插入的占位标记
  总体替换：onerror→onload转换（可选）

F. 输入框选择器速查
────────────────────
  推荐：document.querySelector('.chatMsgTextarea')
       .querySelector('textarea') || container

G. 分隔符速查
────────────────────
  |  → 分隔子项     茜|目击者|15
  ,  → 分隔条目     茜|目击者|15,真红|嫌疑人|-20
  :  → 分隔名和值   hp:65/100
  /  → 当前值/最大  65/100
  :: → 键值分隔     key::value
  ;; → 条目分隔     key1::val1;;key2::val2

H. replaceString转义速查
────────────────────
  "  → \"     JSON字符串内的双引号
  \  → \\     反斜杠
  '  → '      单引号不需要转义
  <  → <      HTML标签中不需要转义
  < → <  innerHTML字符串中替代<
  \n → \n     JSON中表示HTML换行（允许）
  \\n → \\n    JS字符串中的换行符

  引号层次管理：
    最外层：JSON的双引号 "replaceString值"
    第二层：onerror的双引号 onerror="..."
    第三层：JS中的字符串 → 使用单引号 '...'
    需要双引号时 → 使用 \" 转义

I. 复杂度评估参考
────────────────────
  S级（30+规则/200K+字符）：开局系统+状态栏+地图+音乐+图鉴+侧边栏
  A级（15-20规则/100-150K）：状态栏+开局+侧边栏+全局美化+自动判定
  B级（5-12规则/50-80K）：状态栏+选项+侧边栏
  C级（4-8规则/25-40K）：状态栏+选项+简单侧边栏
  D级（1-4规则/<25K）：简单状态栏或单一功能

J. className压缩安全速查
────────────────────
  ✅ d.className = flag ? 'z-cls on' : 'z-cls';
  ✅ var cn='z-cls'; if(f) cn=cn+' on'; d.className=cn;
  ❌ d.className = 'z-cls' + (flag ? ' on' : '');

K. JSON链式校验速查
────────────────────
  安全断点：opt数组闭合]后 + 当前对象闭合}后 + 逗号,
  危险断点：多关闭一层]}/少一个逗号/引号未闭合
  诊断方式：try-catch + 页面输出错误信息
  错误关键词：
    "after JSON data" → 根对象提前关闭（多了括号）
    "expected ','" → 缺少逗号
    "unterminated string" → 引号未闭合

L. 美化/功能分离速查
────────────────────
  改皮肤 → 改美化层
  改脑子 → 改功能层
  加/减器官 → 两边都改
  分离标记：&继续功能& 或自定义标记

M. 响应式断点速查
────────────────────
  <768px  → 手机（100%宽，max-width:520px）
  ≥768px  → 平板（600-640px + 侧边框）
  ≥1200px → 电脑（680-720px + 圆角 + 阴影）
  ≥1600px → 大屏（760-800px）
  居中强制：.content.left .z-xxx-box { margin:auto !important }

N. 五文件架构速查
────────────────────
  ┌───────┬──────────────────────┬──────────────────┐
  │ 编号  │ 文件类型             │ 平台承载方式     │
  ├───────┼──────────────────────┼──────────────────┤
  │ 1     │ 界面文件             │ <界面>正则规则   │
  │ 2     │ CSS文件              │ <css>正则规则    │
  │ 3     │ 初始化代码文件       │ <初始化>正则规则 │
  │ 4     │ 回复代码文件         │ <回复>正则规则   │
  │ 5     │ 中间态正则列表文件   │ 各自独立正则规则 │
  └───────┴──────────────────────┴──────────────────┘

  statusbar承载：<css><初始化><界面>
  beginning承载：首条消息 + 回复级触发标记
  初始化和回复代码必须分开维护

O. 六步执行链速查
────────────────────
  Stage 1: var self = this;                     — 必须
  Stage 2: waitReady(next)                      — 可选（依赖初始化时）
  Stage 3: runScoped() → closest('.content.left') — 可选（回复级视图）
  Stage 4: isLatestReplyBox()                   — 可选（全局视图更新前）
  Stage 5: runLatestOnly()                      — 可选（全局视图更新）
  Stage 6: removeSelf()                         — 必须

  组合A（回复级）：1 + 3 + 6
  组合B（全局级）：1 + (2) + (3) + 4 + 5 + 6

P. 视图分类速查
────────────────────
  全局单例视图：页面唯一，跨回复复用
    → 地图/侧边栏/弹窗/图鉴/全局面板
    → 更新前必须通过最新回复仲裁

  回复级重复视图：每条回复独立实例
    → 状态栏块/选项块/日志块
    → 优先读取同级中间态，不依赖最新仲裁

Q. 中间态标准链路速查
────────────────────
  原始AI文本 → 正则替换成结构化节点 → 回复代码读取节点
  → 转成标准数据 → 投放到界面

  推荐形式：隐藏div / data-*属性 / 子节点文本 / 多个同级节点
  正则列表只做"正则式+替换"，不混入运行逻辑

R. 触发机制速查
────────────────────
  初始化img：页面打开运行一次 → 执行完自删
  回复img：每条回复各一个 → 执行完自删
  历史回复：页面打开时多个回复img可能同时加载
  新回复：完成生成+正则替换完成后触发
  仲裁依据：.content.left DOM顺序（length-2）

S. 初始化代码速查
────────────────────
  必须包含：
    window.uiReady = false        — 初始化完成标记（末尾设true）
    window.uiApp = {}             — 全局控制对象
    页面定位代码                   — 避免遮挡顶部栏/底部栏
    this.remove()                 — 执行完自删

  不应包含：
    某一条回复的具体业务数据
